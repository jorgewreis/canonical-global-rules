# Canônico de Documentação Técnica

## 1. Finalidade do Documento

Este documento define as regras canônicas obrigatórias para a geração, revisão e padronização de documentação técnica, independentemente do projeto, tecnologia ou área de aplicação.

Ele complementa o Canônico Global, especializando regras de estrutura, forma, profundidade e qualidade específicas para documentos técnicos.

Nenhuma regra aqui pode contrariar o Canônico Global.

## 2. Escopo de Aplicação

Este canônico se aplica, entre outros, aos seguintes tipos de documentos:

- Documentação de arquitetura de sistemas
- Documentação de stack tecnológica
- Documentação de banco de dados
- Documentação de APIs
- Documentação de requisitos e funcionalidades
- Documentação de processos técnicos
- Documentação de padrões e governança técnica

Sempre que o comando envolver documentar, descrever, definir, especificar ou formalizar, este canônico deve ser aplicado.

## 3. Estrutura Obrigatória dos Documentos

Todo documento técnico deve conter, no mínimo, a seguinte estrutura lógica:

### Introdução

Contextualização do documento, seu objetivo e sua importância no projeto.

### Visão Geral Técnica

Descrição do cenário técnico, do problema ou da solução em nível conceitual.

### Seções Técnicas Específicas

Seções variam conforme o tema (ex.: banco de dados, arquitetura, segurança), mas devem ser claramente nomeadas e organizadas.

### Considerações Finais ou Observações Técnicas

Limitações, premissas, decisões relevantes ou pontos de atenção.

Se o documento for extenso, a estrutura pode ser expandida, mas nunca reduzida.

## 4. Regras de Profundidade e Escrita

### 4.1 Parágrafos

Cada seção principal deve conter no mínimo três parágrafos completos.

Cada subseção deve conter no mínimo 2 parágrafos completos.

Parágrafos devem ser explicativos e narrativos, não meramente introdutórios.

Não é permitido substituir parágrafos por listas extensas.

### 4.2 Tópicos e Listas

Listas devem ser usadas somente para:

- exemplos
- checklists
- enumeração objetiva de itens

Listas nunca devem ser o conteúdo principal da seção.

Sempre que houver lista, deve haver texto explicativo antes e depois.

## 5. Linguagem e Tom Técnico

Linguagem técnico-descritiva, clara e precisa.

Evitar termos genéricos sem explicação (“eficiente”, “moderno”, “robusto”).

Não usar linguagem promocional ou marketing.

Não usar emojis, humor ou informalidades.

O texto deve ser compreensível por um profissional técnico, mesmo que não tenha participado do projeto.

## 6. Layout e Organização Visual

Títulos e subtítulos devem seguir hierarquia lógica.

Evitar seções excessivamente longas sem subdivisão.

Manter espaçamento visual claro entre blocos conceituais.

Código, quando necessário, deve ser:

- claramente delimitado
- acompanhado de explicação textual

## 7. Regras de Conteúdo Técnico

Ao documentar qualquer componente técnico, sempre que aplicável, deve-se abordar:

- Finalidade: por que isso existe
- Contexto: onde se encaixa no sistema
- Decisões técnicas: por que foi escolhido
- Impactos: vantagens, limitações e riscos
- Dependências: integrações ou pré-requisitos

Se alguma dessas dimensões não puder ser descrita, isso deve ser explicitado.

## 8. Validação Obrigatória Antes da Entrega

Antes de finalizar um documento técnico, verificar:

- A estrutura mínima foi respeitada?
- Cada seção principal tem ao menos três parágrafos?
- As listas são complementares, não dominantes?
- As decisões técnicas estão explicadas?
- Não há suposições implícitas sem justificativa?

Se algum item falhar, o documento deve ser ajustado.

## 9. Conflito e Hierarquia de Regras

Em caso de conflito, prevalece a seguinte ordem:

1. Canônico Global
2. Canônico de Documentação Técnica
3. Comando específico do usuário

## 10. Regra de Evolução

Este canônico:

- Deve ser versionado
- Deve evoluir apenas quando o padrão de documentação mudar de forma consciente
- Não deve ser alterado pontualmente para “ajustar” uma resposta específica

## 11. Declaração Final

Este documento define o padrão técnico oficial para toda documentação gerada.
Qualquer documento que não siga estas regras deve ser considerado fora do padrão, mesmo que tecnicamente correto.
